# Browser Broker (bbroker) — Claude Context

## What This Project Is

bbroker is a Kubernetes-native browser-as-a-service. When a client connects via WebSocket, bbrokerd creates an ephemeral browser pod, relays CDP traffic directly to it, and deletes the pod on disconnect. No database, no session store — sessions are pods.

**Why it exists**: sockpuppetbrowser scales only vertically (semaphore per container). bbroker gives per-session pod isolation, direct CDP access, and horizontal scaling.

**GitHub**: https://github.com/jlaska/bbroker  
**Current release**: v0.1.0

---

## Architecture

```
Client (Playwright/Puppeteer/CDP)
  |
  | ws://bbrokerd:4444/cdtp/chrome[?headful=true]
  v
bbrokerd (stateless Go proxy, 2+ replicas)
  - Creates pod via k8s API
  - Watches pod until Ready
  - Discovers CDP via /json/version (or direct WS for proxy images)
  - Bidirectional WebSocket relay
  - Deletes pod on disconnect
  |
  v (direct pod IP, no per-pod Services)
Browser Pod
  ├── browser container  (chromedp/headless-shell OR bbroker-chrome)
  ├── warden sidecar     (bbroker-warden: single-session, idle timeout, self-terminate)
  └── xvfb sidecar       (bbroker-xvfb: only when ?headful=true)
```

### Key Design Decisions (don't revisit without good reason)

- **Stateless**: Kubernetes IS the session store. Pod name = session ID. Any proxy replica can route to any pod.
- **No CRDs for sessions**: BrowserSession CRDs were considered and rejected. Too much state to manage, pods are sufficient.
- **Warden self-terminates**: The warden calls k8s API to delete its own pod on idle timeout, independent of the proxy. Pods clean themselves up even if bbrokerd crashes.
- **Direct pod IP**: No per-pod Services. Pod IPs are routable within the cluster.
- **Alpine runtime base**: distroless (gcr.io) is unreachable from Docker BuildKit on macOS due to Docker Desktop routing constraints for Google IPs. Alpine 3.20 from Docker Hub is equivalent and reliable.

## Browser Modes

### Headless (default, `?headful=false`)
- Browser image: `chromedp/headless-shell:latest`
- Args: nil (image manages its own entrypoint — Chrome on port 9223, socat on 9222)
- Pod containers: browser + warden

### Headful (`?headful=true`)
- Browser image: `ghcr.io/jlaska/bbroker-chrome:v0.1.0`
- Args: generated by `BuildChromeArgs(true, params)` — no `--headless`, explicit CDP flags
- Pod containers: browser + warden + xvfb
- Chrome binding fix: Debian Chromium binds DevTools to 127.0.0.1 even with `--remote-debugging-address=0.0.0.0`. The `bbroker-chrome` entrypoint uses socat (`0.0.0.0:9222 → localhost:9223`) identically to how chromedp/headless-shell works.
- Xvfb X11: containers in a k8s pod share the Linux network namespace, so Xvfb's abstract socket (`:99`) is accessible between containers without shared volumes.
- Race condition: `bbroker-chrome/entrypoint.sh` polls `xdpyinfo` for up to 10s before launching Chrome.

### Direct WebSocket (`--direct-ws-port=3000`)
- For proxy images like sockpuppetbrowser that manage Chrome internally.
- bbrokerd skips `/json/version` discovery and connects directly to `ws://pod_ip:3000`.
- Query params (including `?headful=true`) are forwarded to the proxy image.

## Development Workflow

```bash
# Build binaries
make build

# Run tests
go test ./...

# Integration test (local Chrome, no k8s)
go test -tags=integration ./test/integration/

# End-to-end on kind
kind create cluster --config test/kind-config.yaml
make docker-build REGISTRY=ghcr.io/jlaska VERSION=latest
for img in bbroker-proxy bbroker-warden bbroker-xvfb bbroker-chrome; do
  kind load docker-image ghcr.io/jlaska/${img}:latest --name bbroker-test
done
kind load docker-image chromedp/headless-shell:latest --name bbroker-test
kubectl --context kind-bbroker-test create namespace bbroker-system
helm install bbroker charts/bbroker \
  --kube-context kind-bbroker-test --namespace bbroker-system \
  --set image.proxy.pullPolicy=IfNotPresent \
  --set browser.imagePullPolicy=IfNotPresent \
  --set browser.image=chromedp/headless-shell:latest \
  --set browser.headfulImage=ghcr.io/jlaska/bbroker-chrome:latest \
  --set imagePrepuller.enabled=false --set replicaCount=1
kubectl --context kind-bbroker-test -n bbroker-system port-forward svc/bbrokerd 4444:4444 &
python3 test/e2e_test.py
kind delete cluster --name bbroker-test

# Release
git tag vX.Y.Z && git push origin vX.Y.Z
# CI builds and pushes multi-arch images on tag push
# For local builds, pre-pull base images then:
docker pull --platform linux/amd64 alpine:3.20 && docker pull --platform linux/arm64 alpine:3.20
docker buildx build --platform linux/amd64,linux/arm64 -f Dockerfile.proxy \
  -t ghcr.io/jlaska/bbroker-proxy:vX.Y.Z --push .
```

## Known Issues / Gotchas

- **buildx + Docker Desktop on macOS**: Google IPs (`142.251.x.x` / gcr.io) are unreachable from inside the BuildKit daemon even with `--driver-opt network=host`. Use Docker Hub images for base layers. Alpine works; distroless does not.
- **chromedp/headless-shell args**: This image has its own entrypoint that manages Chrome on port 9223 with socat on 9222. Do NOT pass `--remote-debugging-port` or `--remote-debugging-address` args — they conflict. `BrowserArgs` is nil for this image.
- **imagePullPolicy**: Browser pods default to `IfNotPresent`. For kind testing, always pre-load images with `kind load` — Kubernetes defaults to `Always` for `:latest` tags even with `IfNotPresent` set on the pod spec if the image hasn't been loaded.
- **Warden has dead code**: `handleWS` in `internal/warden/warden.go` accepts a WebSocket and proxies to Chrome, but bbrokerd currently connects directly to Chrome's CDP port. The warden's WebSocket is unused in the production path. Future work: route through the warden to avoid `--remote-debugging-address` issues.
